# Что осталось реализовать для идеального выполнения пункта 0

Актуально после проверки проекта и прогона `cmake --build build --target run_all_tests -j 4`: все текущие тесты проходят. Уже реализованы и не перечисляются как проблемы: базовые DDL/DML, `int`/`string`, `NOT NULL`, `INDEXED`, JSON для `SELECT`, batch/interactive режим основного CLI, восстановление каталога и таблиц после перезапуска, `database.table`, `BETWEEN`, `LIKE`, сравнения, B-tree индекс, базовый gRPC клиент-сервер, информативные ошибки lexer/parser/semantic/runtime.

## 1) Надежность и целостность таблиц/индексов на диске
- [ ] Проверить атомарность связки "запись строки в `.tbl` + запись ключей в `.idx`".
- [ ] Для `INSERT`, `UPDATE`, `DELETE` гарантировать откат уже примененных изменений при ошибке на середине операции.
- [ ] Добавить тесты на целостность после длинных серий `INSERT/UPDATE/DELETE` с последующим перезапуском `Dbms`.
- [ ] Проверить и описать поведение при ошибках файловой системы: невозможно создать файл, записать страницу, прочитать поврежденную страницу.
- [ ] Убедиться, что индексы после рестарта полностью согласованы с таблицами.

**Критерий готовности:** после любой штатной операции и после перезапуска таблицы и индексы не расходятся.

## 2) Индексная оптимизация: доказуемое покрытие всех подходящих случаев
- [ ] Закрепить тестами использование индекса для `==`, `<`, `<=`, `>`, `>=` по индексируемым `INT`, `STRING`, `BOOL`.
- [ ] Добавить отдельные тесты для fallback на full scan, когда индекс неприменим: `!=`, неиндексируемая колонка, сравнение колонка-колонка, неподходящий тип литерала.
- [ ] Рассмотреть индексный путь для `BETWEEN` по индексируемой колонке, а не только full scan.
- [ ] Добавить простой диагностический механизм для тестов, чтобы можно было проверять, был ли реально использован индекс, без хаков с повреждением header.

**Критерий готовности:** для индексируемых условий поиск идет через индекс, а fallback происходит только когда это ожидаемо.

## 3) gRPC-клиент: интерактивный режим уровня основного CLI
- [ ] Поддержать многострочные SQL-команды до `;` в interactive gRPC client.
- [ ] Поддержать несколько команд в одном вводе, корректно разделяя их по `;` вне строковых литералов.
- [ ] Поддержать `exit` и `quit` без обязательной точки с запятой.
- [ ] Добавить smoke/e2e-тест или тестовый скрипт для gRPC server + client.

**Критерий готовности:** демонстрация клиент-серверного режима не требует ручного правила "одна команда строго в одну строку".

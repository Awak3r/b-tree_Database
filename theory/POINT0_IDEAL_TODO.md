## 1) Надежность `.tbl` + `.idx`

Фактическая модель хранения:

- `data_root` берется из `DBMS_DATA_ROOT`, иначе используется временная директория `coursework_dbms_data`.
- Каждая БД - отдельная директория.
- Каждая таблица - файл `<table>.tbl`.
- Page `0` в `.tbl` хранит `TableHeader` и схему колонок.
- Записи лежат на страницах `1..next_page_id-1`.
- Физический адрес записи - `Rid{page_id, slot_id}`.
- Удаление записи логическое: slot помечается пустым через `size = 0`, уплотнение страниц не делается.
- Индекс колонки - файл `<table>__<column>.idx`.
- `.idx` хранит B-tree страницы: ключ индексируемой колонки + `Rid` записи в `.tbl`.

Честные гарантии:

- [x] После успешных штатных операций `.tbl` и `.idx` согласованы.
- [x] После обычного перезапуска `Dbms` данные, схемы и индексы читаются из файлов.
- [x] `INSERT` переписан на более атомарный путь: batch валидируется заранее, страницы готовятся в памяти, индексы применяются через rollback, header таблицы пишется последним.
- [x] `INSERT` покрыт тестами на частичный batch failure и отказ записи второго индекса.
- [x] `UPDATE` покрыт тестами на duplicate indexed key, успешный restart, отказ второго индекса с откатом первого.
- [x] `DELETE` покрыт тестами на restart, очистку indexed key и отказ второго индекса с откатом первого.
- [x] Длинная серия `INSERT/UPDATE/DELETE` после restart сверяет full scan, indexed lookup и `Rid -> .tbl`.
- [x] Несколько БД с одинаковыми таблицами не смешивают `.idx`.
- [x] Удаленный `.idx`, обрезанный `.tbl` и битый `Rid` дают runtime error, а не тихо неверный результат.

Честные ограничения:

- [ ] Нет WAL/journal/copy-on-write.
- [ ] Нет гарантии восстановления после `kill -9` или отключения питания между page writes.
- [ ] Нет ACID-транзакций между несколькими SQL-командами.
- [ ] Нет модели конкурентных писателей.
- [ ] Отказы записи `.idx` покрыты через реальную ошибку пути: файл индекса заменяется директорией.
- [ ] Отказы записи `.tbl` без искусственных хуков пока не покрыты автоматическим тестом: `chmod` под root ненадежен, а production-код не должен содержать test-only логику.
